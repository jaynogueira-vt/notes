Proposal · Architecture

The Software Factory, designed for change

This proposal sets out an org-wide software factory: one platform that turns engineering work into reviewed, merge-ready pull requests, architected so every layer stays best-of-breed and replaceable as the AI landscape shifts. It states the recommended architecture, the build-versus-buy decision for each layer, and the decision requested of leadership.

From: Jay Nogueira To: Engineering leadership Status: Draft · for discussion

Decision requested

Approve the contract-first hybrid architecture described here: an organization-owned control plane, a multi-vendor swap plane for the fast-moving AI layers, and the staged rollout below. Specifically, authorize Phase 0 and Phase 1 with a single team to produce real merge-rate data before any further commitment.

01 · The problem

Routine engineering work is bounded by headcount, and AI tooling is unmanaged

A large share of engineering throughput goes into changes that are well understood but still fully manual: bug fixes with a clear reproduction, dependency and framework upgrades, feature-flag removals, small well-specified features, and mechanical refactors. This work is necessary and rarely difficult, yet it scales only with the number of engineer-hours available to write it and shepherd it through review. More capacity means more headcount.

At the same time, AI coding tools have become individually capable but are adopted ad hoc, one engineer at a time, with no shared standards for isolation, review, evaluation, or audit. At the organization level this is neither safe nor measurable: there is no consistent quality gate, no record of what a tool changed or why, and no data on whether any of it improves delivery. The two problems compound. The work that is most automatable is still done by hand, while the automation that does exist runs outside any controls.

What this proposal solves

Turn that class of work into a standardized, inspected, auditable production line, and put AI tooling behind organization-owned controls rather than individual discretion. One line, one quality gate, one audit trail, available to every team.

02 · What it is

A production line that turns a unit of work into a reviewed, merge-ready pull request

The factory accepts a unit of work (a ticket, a flag to remove, a dependency to bump) and normalizes it into a structured work item with explicit scope and a risk class. A single run then proceeds end to end, and a person makes the final call.

1

Intake

A ticket or request becomes a structured work item: what to change, where, the acceptance criteria, and how risky it is.

2

Build

An agent plans the change and writes it in an isolated, ephemeral workspace with scoped, short-lived credentials.

3

Inspect

The full quality suite runs against the result: tests, lint, type-check, security scan, build. Nothing proceeds on a failed gate.

4

Deliver

The system opens a pull request carrying the diff and the evidence of every check, and links it back to the originating ticket.

5

Review and merge

A human reviewer approves, requests changes, or rejects. On approval, a human merges. The factory prepares and routes; it never merges on its own.

Every step (the plan, the tool calls, the diffs, the decisions, the retries) is recorded, so any run can be reconstructed from work item to merged code. The detailed station-by-station architecture is in the second half of this document; the end-to-end flow diagram there traces one run through every layer.

What changes

A draft, not a blank editor

Routine work arrives as a pre-validated pull request to review, instead of a task to write from scratch.

What changes

Throughput beyond headcount

This class of work stops scaling purely with engineer-hours. Engineering time shifts to reviewing and deciding.

What changes

One gate, full audit

Every AI-produced change runs through the same controls and leaves a complete record, instead of ad-hoc tooling with none.

Part II

The recommended architecture

The rest of this document is how the line is built: the design principle, the nine layers and their contracts, the build-versus-buy decision for each, the vendor landscape, and the staged rollout. The aim throughout is a line that stays best-of-breed as the AI tools underneath it change.

03 · Design for change

The line is built to be replaced, because the AI layer changes constantly

The central argument is straightforward: the components that draw the most attention (the coding agent, the model, the sandbox) are the fastest-depreciating assets in the stack. A frontier coding agent stays useful for a few quarters before something better ships. The platform should therefore not be bet on any single one of them. The durable approach is to own the stable contracts between layers and treat the components as replaceable parts behind those contracts. With that in place, build-versus-buy stops being a one-time bet and becomes a per-layer decision that can be re-made every quarter without re-architecting the line.

The principle this proposal is built on

Build and own the contracts and the org-specific stations; source every commodity station from the best vendor available today and replace it when a better one ships. That single discipline is what lets the platform survive a market that reshuffles every few months.

Constraint

The AI layer churns fast

The best coding agent, the best model, and the best sandbox each change on a months-long cadence. Any architecture that hard-wires one of them is obsolete on arrival.

Requirement

Org-wide and serious

This must serve every team with isolation, governance, observability, and SLAs. Not a trial, not a single-team script. Professional infrastructure.

Objective

Best, not cheapest

The selection criterion is engineering excellence and long-horizon optionality, not the lowest bill or the shortest path. Optionality is the feature being purchased.

04 · The bar

The eight properties this platform must hold to

"Software factory" borrows the discipline of manufacturing: a standardized line, repeatable stations, inspection at every step, and full traceability. The platform should be judged against the eight properties below. A system that does not meet all eight is a collection of scripts, not a platform, and should not ship as one.

01

Standardized line, stable stations

Work flows through the same defined stations every time. Each station exposes a versioned contract, so what happens inside it can change without disturbing its neighbours.

02

Independent, replaceable stations

The defining property. Any station (agent, model, sandbox, orchestrator) can be swapped for a better one, or run in parallel, with no change to the rest of the line. This is what survives the AI churn.

03

Golden paths and self-service

Teams onboard by configuration, not by forking infrastructure. A paved road with sensible defaults; deviation is possible but not required.

04

Inspection at every station

Tests, lint, type, security, and build gate every run. Above that, an offline evaluation harness measures agent yield so component swaps are decided on data, not vibes.

05

Full traceability

Every plan, tool call, diff, decision, retry, and failure is reconstructable. The factory equivalent of a requirements-traceability matrix: from work item to merged code, end to end.

06

Bounded blast radius

Per-run isolation, least-privilege and short-lived credentials, network egress policy, per-team namespaces. One run can never harm another team or production.

07

Measured throughput and yield

Cycle time, merge rate by work-item class, cost per merged PR, review burden. A factory that cannot be measured cannot be improved or defended.

08

Human authority at the gate

The factory prepares and routes; humans approve and merge. Autonomous merge is a separate, later, evidence-gated decision, never the default.

Why a human stays at the gate

Agent productivity is not yet a settled question. The most rigorous public study to date (METR, 2025) found experienced developers were measurably slower with AI tooling while believing they were faster, and agent-authored PRs shift load onto reviewers. This proposal therefore treats agent output as a draft to be inspected, makes the reviewer experience a first-class part of the build, and measures real yield before relying on it. Merge stays human, without exception.

07 · The core idea

The swap plane: one contract, many vendors, decided by data

Layers 4 to 7 are where the AI world moves fastest, so they get the strongest contract discipline. The agent-execution contract (solve(WorkItem, RepoRef) → RunResult) allows several implementers to run behind one interface: route by work-item class, run them head-to-head on the same ticket, and let the eval harness select the winner. When a new frontier agent ships, adoption is a new adapter and an A/B run against the eval suite, not a re-architecture.

Route

By work-item class

Bug fix, multi-file feature, refactor, flag removal. Each class routes to the implementer that wins on it. One factory, many specialists.

Race

Head-to-head

On high-value tickets, run two implementers on the same WorkItem and keep the better PR. A configuration toggle, not a project.

Replace

Without re-architecture

When a better agent ships, write one adapter, run it against the eval suite, promote it if it wins. The line never stops.

08 · The strategy

Build the moat, buy the commodity, contract everything

The build-versus-buy decision is made per layer, on one rule: build what is org-specific and durable; buy what is commodity and fast-moving; put a contract on every boundary so any decision is reversible. No internal team will out-build the frontier agent vendors on the implementer, and none should try. The organization is the only party that can build org-aware intake, knowledge, reviewer experience, and governance.

Why not a single vendor end-to-end?

A single-vendor factory (one platform running the whole line) is the easiest path and a real option, but it fails the two requirements ranked highest: independence and easy to change layers. It imposes the vendor's orchestration, reviewer UX, model choices, and roadmap. When a better agent ships elsewhere, adopting it means leaving the platform. Buying a vendor agent is correct; buying a vendor factory trades away the optionality that is the whole point.

Why not pure in-house?

A capable org can build this. But pure in-house means owning the implementer, the sandbox, and the model plumbing, which are exactly the parts the market improves faster than any internal team can. Pure build fails "best-of-breed as the world shifts." The right amount of build is the moat and the contracts, not the commodity.

10 · Recommendation

Build an owned control plane over a multi-vendor swap plane

The recommendation is to build a thin, durable control plane the organization owns end to end: intake and policy, the reviewer experience inside its own identity, the gates, the eval harness, and the knowledge layer, run on a managed cloud runtime. The implementer, the sandbox, and the models are treated as best-of-breed plug-ins behind contracts; several run at once, and measured yield decides which one wins each class of work. Buy the commodity, build the moat, and keep every boundary swappable. A single-vendor factory and a pure in-house build are both rejected, for the reasons set out in the build-versus-buy and scorecard sections above.

The org builds · the moat

Control plane + knowledge + gates

Intake and scope policy, the authenticated reviewer surface, the quality + eval gates, and retrieval over org systems. Durable, org-specific, and the instruments by which every other choice is made. They do not churn as models and agents are replaced.

The org buys / bundles · the commodity

Agents + sandbox + models + runtime

Multiple vendor implementers behind one contract, a managed agent runtime, sandbox + model bundled with the agent, delivery adapters, and live tools over MCP. Cloud-native, scalable, replaced on the market's cadence.

Decision: a managed runtime, not a self-hosted workflow cluster

The orchestration runtime is the one layer most tempting to over-build. Run it on a managed agent runtime rather than standing up a self-hosted durable-workflow cluster. The line is mostly idle human-review waits, and that wait is better modeled as owned persisted state with scheduled re-checks than as a durable workflow timer. The managed runtime covers the bounded active execution — plan, code, gate — with hosted sessions, isolated containers, retries, and observability. Crucially, the org still owns the orchestration contract, so a durable-workflow engine remains a drop-in replacement behind the same interface if the workflow ever outgrows this. The escape hatch is the point: lean now, swap later, no re-architecture.

12 · Sequencing

Rollout, gated by evidence not calendar

Each phase ends in a gate that must be met before the next begins. Phases are illustrative sequencing, not committed dates; durations and capacity are intentionally left blank for planning.

Phase 0Gate · contracts frozen, eval suite runs

Contracts & the instrument panel

  • Define and version the four swap-plane contracts: WorkItem, solve(), Sandbox, model gateway endpoint.
  • Stand up the eval harness over a labelled set of real tickets, by work-item class. The authority for every later swap.
  • Pick a managed runtime and a sandbox; wire one implementer behind the solve() contract end to end on one repo.
Phase 1Gate · one team, real merge-rate data by class

First team on the paved road

  • One team runs real work items through the full line with authenticated review and human merge.
  • Two implementers behind the contract, A/B by work-item class on the eval suite.
  • Per-run sandbox isolation, least-privilege short-lived credentials, full audit trail demonstrated.
Phase 2Gate · isolation + HA + SLOs validated

Multi-team general availability

  • Onboard teams by namespace + configuration; quotas, rate limits, cost attribution per team.
  • Add a third vendor agent and the open-source fallback to prove the swap plane under load.
  • Define SLOs and on-call; observability and audit meet the professional bar.
Phase 3Gate · self-service onboarding documented

Org platform

  • Self-service onboarding; golden paths with sensible defaults; per-team policy as configuration.
  • Quarterly swap review: re-run the eval suite against new vendors, promote winners by adapter.
  • Evaluate (separately, evidence-gated) automated review and rule-based auto-merge for lower-risk repos.

13 · Risks

The risks, and how they are handled

  • Contract churn. If the swap-plane contracts are wrong, every adapter leaks and the independence promise collapses. Mitigation: keep contracts minimal, version them, and prove them with two real vendors in Phase 1 before committing.
  • Bespoke-orchestrator creep. Running orchestration on a managed runtime instead of a workflow engine means the org owns the workflow state machine. If the line grows complex (parallel fan-out, compensation / rollback, replay-for-compliance), that hand-rolled layer becomes the thing a durable-workflow engine would have given for free. Mitigation: keep the orchestration contract minimal so the engine drops in behind it the moment the hand-rolled version strains.
  • Integration tax. Multi-vendor means maintaining adapters and a thin control plane forever. This is the price of optionality; it is justified only because changeability is the top requirement. If that ever stops being true, collapse to fewer vendors.
  • Agent yield, not orchestration, is the real bottleneck. Public evidence and an honest read of current autonomous-PR merge rates say the implementer is where value is won or lost. If yield stays low across all vendors, a beautifully swappable factory still produces drafts nobody merges. The eval harness exists to catch this early and honestly.
  • Review burden. Agent PRs move load onto reviewers. The reviewer experience must be a first-class build, and merge stays human until data justifies otherwise.
  • Reuse temptation. An existing in-house solver may already implement much of this loop. It is a useful Phase 0/1 reference and possibly the open-source-fallback implementer behind the contract, but the architecture should not be anchored to it: reuse a part only where it is the best option behind the contract, not because it exists.
