UX/UI Design Review

A repository-grounded design-review pipeline for AI coding agents. It captures the current state of a UI surface, audits it against 13 categories, and produces a prioritized findings report with concrete patches that reuse the project's own design tokens — never invented values.

Get the skill (Markdown)

In one paragraph

Most UI regressions are cheap to catch at review time and expensive to catch after ship: a hardcoded hex instead of a token, off-grid spacing, a missing empty state, a score column nobody can explain. This skill is the structured pass that catches them. It is deliberately a review, not a redesign — it reads the codebase, learns the project's existing design language, and reports deviations from that language rather than imposing an outside aesthetic.

What it is

  • A 5-step workflow: establish the project's design DNA → capture the surface (page, spec, or PR diff) → run the 13-category checklist → write a prioritized findings report → hand back.
  • A 13-category audit checklist: visual hierarchy, typography, spacing & alignment, color & WCAG contrast, design-token consistency, loading/empty/error states, interactions, accessibility, responsive behavior, state consistency, pagination edge cases, copy, and metric explainability.
  • An output format that ranks every finding P0 / P1 / P2 with Where, What, Why, and a concrete Fix that uses existing tokens.
  • Agent-agnostic: written for any skill-aware AI coding agent; adapt the file-path examples to your repo.

What it does

  • Greps the repo for tokens, primitives, and hardcoded hex so findings cite real evidence (file:line), not generic best-practice.
  • Reviews a written spec before implementation — its highest-leverage mode — to catch parallel-component invention while the fix is still a one-line edit.
  • Flags opaque metrics (composite scores, non-obvious columns) that ship without an explainer tooltip.
  • Produces ready-to-apply patches for P0 issues, gated on your approval.

What it does not do

  • It does not edit code, run formatters, create tickets, commit, or open PRs on its own — it is read-only until you explicitly ask for fixes.
  • It does not invent new colors, spacings, or radii. New values get questioned, not silently added.
  • It does not import outside conventions — "Stripe does it this way" is not a finding unless your design system already aligns with it.
  • It does not replace a correctness/security code review or functional QA — it runs alongside them, covering the visual/UX layer only.
  • It does not chase pixel-perfection; it focuses on systemic issues, not every 1–2px offset.

When to reach for it

Any UI feature request, "review this screen", "does this look right?", "match the design system", a PR that touches components/CSS/Tailwind, or the end of a planning cycle that produces UI. Run it on the spec before implementation, then again on the implementation before the PR.

The 13 audit categories

  1. Visual hierarchy — one dominant primary action, exactly one h1, consistent elevation.
  2. Typography — match the existing type scale; tabular numerals in comparison columns; purposeful weight.
  3. Spacing & alignment — on-grid increments, consistent panel padding, deliberate vertical rhythm.
  4. Color & contrast — WCAG AA, no hardcoded hex, one token per status, color never the sole signal.
  5. Design-token consistency — every value resolves to a token; no near-duplicate tokens.
  6. Loading / empty / error trinity — every async region has a matching skeleton, an explicit empty state, and an error state with retry.
  7. Interactions — visible affordance states, in-flight disabling, blur-time validation, reduced-motion support.
  8. Accessibility — keyboard reachable, visible focus ring, labelled fields and icon buttons, sequential headings.
  9. Responsive behavior — desktop, tablet, and mobile render without overflow, or degrade gracefully by design.
  10. State consistency — the same backend concept shows the same value everywhere; buttons reflect legal actions.
  11. Pagination edge cases — first/last/middle pages, clamped overflow, visible totals, distinct empty-filter state.
  12. Copy & microcopy — single language, sentence case, actionable error messages.
  13. Metric explainability — every composite score, opaque column, and non-trivial badge carries a concise tooltip that stays in sync with the source of truth.
