Agent Harness Optimization

A method for converging every AI coding agent in a repository on the same accurate rules: a single-source-of-truth AGENTS.md, thin pointers for every tool-specific file, a router pattern for large rule sets, CI that makes the gates blocking instead of advisory, and a zero-cost watchdog that babysits the PRs to merge.

Get the skill (Markdown)

In one paragraph

Most repos that adopt AI coding agents accumulate the same debt: the rules get copy-pasted into CLAUDE.md, .cursor/rules, and AGENTS.md, then drift apart; half the tools (Copilot, Gemini) never get a rules file at all; the rules contradict the repo's own code (a "Vitest only" rule next to a live Jest config); and the "pre-push gates" the rules mandate are enforced by no CI job. This skill fixes all four with one architecture: AGENTS.md is the single source of truth, every tool file is a three-line pointer to it, rule sets past ~400 lines split into a small router plus focused per-domain files so agents load only what their change touches, and the repo's own gates become CI jobs that fail loudly. The delivery shape is one reviewable PR per repo, then a plain shell watchdog (no tokens burned) that keeps each PR current with its base branch, squash-merges on human approval, and wakes the agent only when something needs actual judgment.

What it is

  • A five-point audit per repo: duplication and drift, missing tool coverage, self-contradictions, unenforced gates, stale facts.
  • The SSOT-plus-pointers architecture: all rules in AGENTS.md; CLAUDE.md, GEMINI.md, copilot-instructions.md, and the Cursor .mdc reduced to pointers that cannot drift.
  • The router pattern for big rule sets: a ~90-line AGENTS.md with a "your change touches X, read Y" table routing to focused files under .agents/rules/.
  • CI recipes that make the rules blocking: formatter, build, vet, and test gates that fail loudly, with the quoting and stderr pitfalls already solved.
  • Content rules worth baking in everywhere: comment discipline (no ticket IDs in code comments, no overcommenting), the repo's number-one destructive-mistake risk repeated in every pointer, inline reviewer instructions.
  • A reusable babysit-prs.sh watchdog: updates branches that fall behind, squash-merges on human approval, exits to wake the agent on failures, conflicts, or new review threads.

What it does

  • Works in isolated git worktrees cut fresh from the current origin default, never in your local checkouts.
  • Fixes the violations the rules ban in the same PR (deletes the stale test config, runs the formatter), so the harness never contradicts the code it governs.
  • Verifies everything locally before pushing: .mdc frontmatter parses, CI YAML parses, the repo's real gates run green in the worktree.
  • Cross-applies review feedback: a comment received on one repo's PR that applies to another gets fixed there too, unprompted.
  • Ships one PR per repo and drives each to zero unresolved threads, then hands monitoring to the shell watchdog instead of polling with the model.

What it does not do

  • It does not impose the router on small repos: a 100-160 line AGENTS.md plus pointers is the right size, and splitting it would be ceremony.
  • It does not leave rules as advice: a gate that no CI job runs is a suggestion, and suggestions drift.
  • It does not merge anything without a human approval on record, and only when merge authority was explicitly delegated.
  • It does not trust a stale review state: approval detection uses each human reviewer's latest review and excludes bots.
  • It does not burn model tokens watching CI spinners; the watchdog is plain shell and wakes the agent only for judgment calls.

When to reach for it

"Optimize our AGENTS.md", "standardize agent instructions across repos", "our agent rule files drifted", "add Copilot/Gemini/Cursor coverage", "split this huge AGENTS.md", "make the agent rules blocking in CI". Best fit: a set of repos that adopted AI coding agents organically and now have inconsistent, duplicated, or decorative rule files. Also the shape to copy when setting up the harness in a brand-new repo, so it starts converged instead of drifting into this state.

The discipline, in six rules

  1. One source of truth: rules live in AGENTS.md; every tool file is a pointer. Duplicated text is future drift.
  2. Route, don't bloat: past ~400 lines, a router plus focused rule files beats a monolith that burns the model's context.
  3. Practice what you preach: fix the repo's own violations of its rules in the same PR, or agents learn the rules are decorative.
  4. Enforce in CI: every gate the rules claim becomes a CI job that fails loudly; no stderr suppression, no || true.
  5. Repeat the fatal warning: the repo's number-one destructive-mistake risk goes in every pointer file, not just the SSOT.
  6. Babysit for free: once only human approval is missing, a shell watchdog owns the wait; the agent returns only for judgment.
